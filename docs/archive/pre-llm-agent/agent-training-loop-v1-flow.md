# Agent 环训练 V1 总流程

## 1. 文档目标

本文定义 Agent 环训练 V1 的用户侧流程、运行节奏、托管方式、汇报方式和安全边界。

目标不是做一个“完全自由驾驶”的自动调参系统，而是做一个：

- 轮内安静
- 轮间可控
- 对比优先
- 可完全放权、也可随时接管
- 不依赖 RAG 才能运行

的半自动 / 全自动循环训练器。

---

## 2. V1 核心原则

### 2.1 训练时少打扰
- 训练进行中默认不按 epoch 主动汇报。
- 只在以下节点汇报：
  - 启动成功
  - 一轮结束
  - 异常发生
  - 最终结束

### 2.2 一轮结束后必须过“轮间闸门”
一轮训练结束后，不允许直接裸奔进入下一轮。

必须经过：

1. 收集本轮事实
2. 对比上一轮
3. 对比当前最佳轮
4. 生成下一轮计划
5. 校验是否越过授权边界
6. 再决定自动继续 / 等用户介入 / 停止

### 2.3 对比优先
循环训练的核心价值不是“又跑了一轮”，而是“这一轮相对前一轮 / 最佳轮发生了什么变化”。

### 2.4 知识库只做辅助，不接管控制权
V1 不依赖 RAG。

知识库在 V1 的角色：

- 辅助解释
- 补充候选建议
- 提供规则来源

控制权仍然来自：

- 训练事实
- 硬编码边界
- 规则引擎
- 用户授权

---

## 3. 核心概念

### 3.1 Loop
一次完整的 Agent 环训练任务，包含多轮训练。

### 3.2 Round
Loop 里的单轮训练。

### 3.3 Best Round
当前 Loop 内表现最好的轮次。

### 3.4 托管级别
决定 Agent 有多大自治权。

### 3.5 授权边界
即使是全托管，也必须在明确边界内运行。

---

## 4. 用户可见配置

V1 用户侧只暴露 4 组配置，不要求用户理解实现细节。

### 4.1 托管级别

#### 审阅模式
- 每轮结束都停
- 下一轮必须等用户确认

#### 保守自动
- 小改动自动继续
- 大改动 / 异常 / 依据不足时停下来等用户
- 作为默认模式

#### 全托管
- 在授权边界内自动跑完整个 Loop
- 不需要用户逐轮介入
- 只在异常 / 结束时汇报

### 4.2 停止条件
至少支持：

- 最大轮数
- 目标指标阈值
- 连续无提升轮数
- 最大失败次数
- 可选最大运行时长

### 4.3 可调范围
V1 默认只开放以下参数自动调整：

- `lr0`
- `batch`
- `imgsz`
- `epochs`

默认不自动做的大改动：

- `optimizer` 切换
- `resume` 策略切换
- 大范围 augment 策略切换
- 未授权的超范围改动

### 4.4 汇报级别

- 简洁：启动 / 轮结束 / 异常 / 最终结束
- 标准：简洁 + 更完整的轮次对比卡
- 留档：标准 + 自动导出最终报告文件

---

## 5. 启动流程

### 5.1 用户发起
用户用自然语言发起，例如：

- “开一个保守自动的环训练，最多 5 轮”
- “开一个全托管环训练，目标 mAP50 到 0.8”
- “这个数据集跑自动复训，每轮结束给我看对比”

### 5.2 系统解析最少必要信息
至少明确：

- 数据集
- 基模型
- 运行位置（本机 / 指定服务器）
- 托管级别
- 停止条件

### 5.3 自动预检
启动前自动做：

- readiness
- prepare（必要时）
- training_preflight

### 5.4 生成计划卡
启动前给用户一张计划卡，只展示最重要信息：

- 训练位置
- 数据集
- 基模型
- 托管级别
- 最多跑几轮
- 停止条件
- 可调范围
- 汇报级别

### 5.5 首次确认
启动前只确认一次，不要求用户逐项填技术参数。

确认后：

- 生成 `loop_id`
- 进入后台运行

---

## 6. 单轮生命周期

每轮固定经过以下阶段：

1. 准备本轮参数
2. 启动训练
3. 训练进行中（默认静默）
4. 训练结束 / 失败 / 被终止
5. 收集本轮结果
6. 对比上一轮
7. 对比最佳轮
8. 生成下一轮候选计划
9. 过“轮间闸门”
10. 自动继续 / 等用户 / 停止

---

## 7. 轮间闸门

轮间闸门是 V1 的核心安全机制。

### 7.1 必做检查
- 本轮是否有足够事实
- 本轮相对上一轮是提升、下降还是震荡
- 本轮是否刷新最佳轮
- 下一轮参数变更是否越界
- 是否触发停止条件
- 是否触发异常恢复策略

### 7.2 闸门输出结果
只能落到三种结果之一：

#### 自动继续
符合托管级别和授权边界，且无异常。

#### 等用户介入
例如：

- 变更超出白名单
- 理由不足
- 连续失败后还要继续
- 大幅调整关键参数

#### 停止
例如：

- 达到目标
- 连续无提升
- 超预算
- 失败过多
- 用户要求停止

### 7.3 知识建议分流规则

V1 不让知识建议直接接管 loop，但会参与轮间闸门分流。

#### 直接停止
以下建议默认直接停住：

- `fix_data_quality`
- `collect_metrics_first`
- `inspect_labels_and_mapping`

这些场景说明当前更缺事实或更像数据问题，不适合继续自动复训。

#### 进入知识审阅闸门
以下建议默认先进入“分析优先”分流：

- `run_error_analysis`
- `inspect_missed_samples`
- `inspect_false_positives`

这类建议说明当前更适合先理解错检/漏检，而不是盲目大调。

#### 按托管级别决定是否继续
- `review`：始终等待用户
- `conservative_auto`：停在 `awaiting_review`
- `full_auto`：记录分析型建议后，在授权边界内允许继续下一轮

也就是说：

> 同样是“先做误差分析”，在保守自动下会先等你，在全托管下可以继续跑，但会把这条分析结论保留下来。

---

## 8. 运行中的控制动作

### 8.1 查看状态
支持：

- 查看 Loop 当前状态
- 查看当前轮详情
- 查看指定轮详情
- 查看轮次对比

### 8.2 暂停
语义：当前轮可以跑完，但不再自动进入下一轮。

### 8.3 恢复
语义：从暂停状态继续进入后续轮次。

### 8.4 立即终止
语义：立即中止当前训练进程，并结束整个 Loop。

### 8.5 一轮结束后停住
比“暂停”更贴合使用场景。

语义：当前轮跑完后停住，不自动开下一轮。

### 8.6 用户接管
支持用户中途接管，例如：

- “从下一轮开始我自己接管”
- “停在这一轮结束，我手动继续”

---

## 9. 汇报设计

V1 的汇报分三层。

### 9.1 对话内短汇报（默认）
默认只在以下节点汇报：

- 启动成功
- 一轮结束
- 异常
- 最终结束

### 9.2 后台结构化持久化（必须有）
系统后台必须自动保存：

- loop 元信息
- 当前轮次
- 每轮参数
- 每轮结果
- 最佳轮
- 停止原因
- 下一轮计划

### 9.3 最终汇总
结束时在对话里给总汇总。
如用户需要，再导出 Markdown / JSON 文件。

---

## 10. 每轮对比卡

每轮结束时，默认给出一张简短对比卡：

### 10.1 固定结构

#### 事实
- 第几轮结束
- 本轮关键指标
- 本轮 vs 上一轮
- 本轮 vs 最佳轮
- 是否刷新最佳轮

#### 判断
- 当前趋势判断
- 为什么认为该继续 / 该停 / 该谨慎

#### 动作
- 下一轮准备改什么
- 为什么这么改
- 是否需要用户介入

### 10.2 目标
让用户一眼能看懂：

- 发生了什么
- 为什么这么判断
- 下一步会做什么

---

## 11. 最终总结页

Loop 结束后，至少给出：

- 最佳轮
- 最佳指标
- 最佳模型路径
- 每轮关键变化概览
- 停止原因
- 整体趋势判断
- 下一步建议

如果是留档级别，还应支持导出：

- Markdown 报告（人看）
- JSON 结果（系统读）

---

## 12. 授权边界

即使是全托管，也必须有边界。

V1 建议最少包含：

- 最大轮数
- 最大失败次数
- 可选最大运行时长
- 允许调整的参数集合
- 单次允许的最大改动幅度
- 是否允许自动 OOM 恢复
- 是否允许自动提前停止

---

## 13. 异常处理体验

异常不能只报错，必须给出处理结论。

例如：

- 发生了什么
- 系统是否已自动处理
- 是否还会继续
- 是否需要用户介入

典型异常包括：

- CUDA OOM
- 训练进程退出
- 远端断连
- 结果目录不可解析
- 训练长期卡住

---

## 14. 知识库 / RAG 在 V1 的位置

### 14.1 V1 不依赖 RAG
V1 的主链不以 RAG 可用为前提。

### 14.2 知识库在 V1 的职责
- 解释结论
- 补充候选建议
- 为规则提供来源

### 14.3 不让知识库做的事
- 直接接管轮间控制
- 绕过安全边界自动做高风险决策
- 在事实不足时强行给结论

---

## 15. V1 默认体验建议

### 15.1 默认托管级别
保守自动

### 15.2 默认停止条件
- 最大 5 轮
- 连续 2 轮提升不足则停止
- 失败 2 次则停止

### 15.3 默认可调范围
- `lr0`
- `batch`
- `imgsz`
- `epochs`

### 15.4 默认汇报
- 启动时
- 每轮结束时
- 异常时
- 最终结束时

### 15.5 默认运行模式
- 轮内静默
- 轮间过闸门
- 对比优先
- 最佳模型自动锁定

---

## 16. V1 范围

V1 先只做：

- 单机 / 单服务器
- 单任务串行 Loop
- 单 active round
- 后台持久化状态
- 托管级别
- 轮次对比
- 自动停止条件
- 可选远端运行

V1 暂不追求：

- 多机多任务调度
- 大规模超参搜索
- 自由 LLM 调参
- 依赖 RAG 的核心控制链

---

## 17. 建议的实现起点

V1 推荐落在服务端，而不是主要挂在聊天客户端。

建议新增：

- `agent/server/services/training_loop_service.py`
- `agent/server/tools/training_loop_tools.py`

建议对外提供：

- `start_training_loop`
- `check_training_loop_status`
- `inspect_training_loop`
- `pause_training_loop`
- `resume_training_loop`
- `stop_training_loop`

建议内部复用现有能力：

- `training_readiness`
- `prepare_dataset_for_training`
- `training_preflight`
- `start_training`
- `check_training_status`
- `summarize_training_run`
- `analyze_training_outcome`
- `recommend_next_training_step`
- `list_training_runs`
- `compare_training_runs`
- `select_best_training_run`

---

## 18. 当前结论

Agent 环训练 V1 的正确方向不是“做一个自由驾驶的 AutoML”，而是：

> 做一个带托管级别、轮间闸门、对比优先、默认安静、可完全放权也可随时接管的循环训练控制器。
