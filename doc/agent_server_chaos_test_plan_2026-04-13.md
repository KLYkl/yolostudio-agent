# YoloStudio Agent 服务器端聊天 Agent 暴力测试计划（100 条）

> 目标：先把服务器端聊天 Agent 的“变态用户场景”测例库写完整，再逐批落成自动化回归。
> 范围：重点打服务器端聊天 Agent 的对话理解、工具调度、训练控制、状态恢复、异常处理，不先纠结 UI 细节。

---

## 1. 这批测试怎么和 Agent 交互

这批测试的主交互方式不是“靠文件和 Agent 交流”，而是：

- **同一会话内的多轮对话重放**
- **服务器端聊天 Agent 真正接收用户话术**
- **每轮检查 tool 路由、确认层、状态写回、最终答复**

文件在这里的作用只是：

- 存测试语料
- 存预期行为
- 存回归结果
- 存失败样本

也就是：

> **对话是执行通道，状态是骨架，工具结果是依据，文件只是测试语料和留痕。**

---

## 2. 每条测试至少要检查什么

每条测试默认检查 6 件事：

1. **有没有走对主线**
2. **有没有调错 tool / 漏调 tool**
3. **该确认的时候有没有确认**
4. **不该直接执行的时候有没有乱执行**
5. **状态有没有被污染或丢失**
6. **最终回复是不是 grounded，而不是瞎编**

默认还要记录：

- `expected_tool_path`
- `expected_confirmations`
- `expected_state_updates`
- `must_not_do`
- `follow_up_checks`

---

## 3. 优先级定义

- **P0**：必须优先做，容易造成真实误操作或主线崩坏
- **P1**：高价值边界场景，应该尽快补齐
- **P2**：增强型鲁棒性场景，可分批跟进

## 4. 100 条暴力测试清单

## A. 信息缺失 / 模糊输入（C01-C10）

| ID | 优先级 | 用户场景 | 关键预期 |
|---|---|---|---|
| C01 | P0 | “帮我开始训练。”什么都不给 | 不能直接开训；先追问数据集或给出所缺信息 |
| C02 | P0 | “用这个数据集训练。”只给目录，不给模型 | 先进入计划草案；不能默认偷偷开训 |
| C03 | P0 | “用 yolov8n 训练。”只给模型，不给数据集 | 明确缺少数据输入；不能调用 start_training |

| C06 | P1 | 给一个文件路径但说“这是数据集” | 应识别路径类型异常；不能当数据集根直接训 |
| C07 | P1 | 给一个视频目录却说“直接训练” | 应提示先抽帧/准备，不应直接走训练 |

| C24 | P0 | 训练进行中，用户说“换另一个数据集重新训” | 应明确这是新 run，不是当前 run 改参数 |

| C30 | P2 | 训练进行中，用户说“开始第二个不同模型实验” | 应进入并行实验/新 run 讨论，而不是偷停当前 run |

## E. 状态追问 / 结果追问乱序（C41-C50）

| ID | 优先级 | 用户场景 | 关键预期 |
|---|---|---|---|
| C41 | P0 | 没有训练在跑，用户问“训练到第几轮了” | 应明确当前无活动训练 |
| C42 | P0 | 训练刚停，用户马上问“跑完了吗” | 应返回 stopped，不误判 completed |
| C43 | P0 | failed 后问“效果怎么样” | 应走 summarize/analyze，不能装作成功 |
| C44 | P1 | completed 后问“下一步先补数据还是调参数” | 应基于真实汇总与知识层给建议 |
| C45 | P1 | stopped 后问“是不是已经收敛了” | 应保守，不把 stopped 当 final success |
| C46 | P1 | 用户连续问“失败了吗/结束了吗/现在状态呢” | 路由都应稳定到 status |
| C47 | P1 | 用户先问建议，再问“你基于哪次训练说的” | 应能追溯到对应 run |
| C48 | P2 | 用户问“刚刚那次和上次比哪个好” | 应跳 compare_training_runs |
| C49 | P2 | 用户问“最近哪次最值得参考” | 应跳 select_best_training_run |
| C50 | P2 | 用户问“刚才你说数据有问题，依据是什么” | 应引用 summarize/compare/knowledge 信号，不空口编 |

## 6. 这批测试写完后怎么做

这份清单写完后，后续执行方式固定为：

1. **先把 100 条按批次转成自动化对话测试**
2. **每批都要带 tool path / state / confirmation / grounded 断言**
3. **先本地，再 proto，再远端**
4. **真实远端场景只在需要时补最小充分样本**

也就是：

> **先有完整测试语料，再逐批实现自动化，不反过来边想边测。**
