# YoloStudio Agent 支线推进计划（数据治理增强 + 实时预测预研）

> 日期：2026-04-13
> 适用范围：`D:\yolodo2.0\agent_plan`
> 定位：**作为当前主线的支线继续推进**，在不破坏现有训练 / prediction / knowledge 主链的前提下，逐步补齐缺失工具能力。

---

## 1. 当前判断

当前主线已经具备继续开支线的条件：

- server agent 的 `C01-C100` live 对话暴力测试已经全量通过
- 训练主线已经跑通：
  - readiness
  - prepare / preflight
  - start / status / stop
  - summarize / analyze / recommend
- prediction 主线已经跑通：
  - 图片 / 图片目录 / 视频 / 视频目录预测
  - 结果汇总
- 数据提取链已经可用：
  - 图片抽取
  - 视频扫描
  - 视频抽帧

所以现在不是“主线还没站稳”，而是：

> **主线已基本收口，可以开一条受控支线，补齐缺失能力。**

---

## 2. 这条支线要解决什么问题

这条支线不再围着“能不能训练、能不能预测”打转，而是补当前 Agent 还没有接进来的真实能力。

用人话说，当前缺口主要是三类：

### A. 数据治理编辑类
也就是“批量改数据、批量修标签、批量清理脏数据”。

当前还没接进 Agent 的典型能力：

- 批量改数据格式
- 批量改标签内容
- 清理孤儿标签
- 批量补空标签 / 缺失标签
- 按类别整理数据

### B. 预测结果管理类
也就是“预测跑完后，怎么导出、整理、继续处理结果”。

当前还没做厚的部分：

- 导出预测报告
- 统一管理 prediction 输出目录
- 导出结果路径清单
- 关键帧 / 结果文件整理

### C. 实时输入源预测类
也就是“不是拿图片/视频文件，而是直接接实时源”。

当前还没接的部分：

- 摄像头预测
- RTSP 视频流预测
- 屏幕内容预测

---

## 3. 支线原则

这条支线必须服从主线，不允许为了扩功能把主线搞坏。

### 3.1 总原则

1. **先低风险，再高风险**
   - 先做只读或可预览能力
   - 再做会修改数据的能力
   - 最后做实时流和长生命周期功能

2. **先 preview，再 execute**
   - 任何带破坏性的批量改动，都必须优先提供预览
   - 不允许一上来直接暴露“批量改标签/清文件”的裸执行

3. **先 headless 低耦合，再 GUI 强耦合**
   - 优先接 `core/` 中纯 Python、无 UI 生命周期依赖的能力
   - 摄像头 / 屏幕 / RTSP 放后面

4. **每加一批功能，都必须过回归门**
   - 不只做工具级测试
   - 还要跑聊天路由 / 远端 smoke / 必要的 live 对话回归

### 3.2 不做的事情

这条支线当前不做：

- 大规模重写主链结构
- 推翻现有训练计划草案层
- 引入向量库 / embedding / 多代理协作
- 先做复杂 GUI 专属能力再补 headless 工具

---

## 4. 支线结构

这条支线拆成 3 个阶段，按优先级依次推进。

---

## Phase A：数据治理增强（第一优先）

### 目标
把 Agent 从“能检查数据、能训练”推进到：

> **能安全地预览和执行批量数据治理动作。**

### A1. 数据格式转换

用户能直接说人话：

- 把这批标注转成另一种格式
- 先预览会改哪些文件
- 确认后再真正执行

对应能力：

- 预览数据格式转换
- 执行数据格式转换

### A2. 标签批量修改

用户能直接说人话：

- 把某个类别整体改编号
- 删掉某类标签
- 批量修一组标签
- 先看影响范围，再决定是否执行

对应能力：

- 预览标签修改
- 执行标签修改

### A3. 标签垃圾清理

用户能直接说人话：

- 把没有图片对应的标签清掉
- 清理脏标签

对应能力：

- 清理孤儿标签

### A4. 缺失标签兜底生成

用户能直接说人话：

- 给没有标签的图片先补空标签
- 让数据集至少进入可继续处理的状态

对应能力：

- 生成空标签
- 补齐缺失标签

### A5. 按类别整理数据

用户能直接说人话：

- 把含某类目标的图片整理出来
- 按类别分桶，方便抽样和检查

对应能力：

- 按类别整理数据

### A 阶段交付要求

每项能力都必须包含：

- 服务层封装
- MCP tool 暴露
- grounded 输出
- 聊天路由
- 工具级测试
- 至少一条远端 smoke

### A 阶段验收标准

完成后应达到：

- 用户能先预览，再确认执行
- 所有批量写操作都有风险提示
- 对话里不会误把“预览”执行成“真改”
- 远端 `yolostudio` 上能完成最小闭环验证

---

## Phase B：预测结果管理增强（第二优先）

### 目标
把 prediction 从“能跑”推进到：

> **能整理、导出、复用结果。**

### B1. 预测报告导出

用户能直接说人话：

- 把这次预测结果导成报告
- 给我一个可复查的汇总文件

### B2. 输出目录管理

用户能直接说人话：

- 这次预测结果保存在哪
- 帮我整理输出目录
- 导出结果路径清单

### B3. 结果二次整理

用户能直接说人话：

- 把含目标的帧单独列出来
- 只看有命中的结果
- 按类别整理预测产物

### B 阶段交付要求

每项能力都必须包含：

- 统一输出结构
- 可直接被 `summarize_prediction_results` 复用
- 工具级测试
- 远端 smoke

### B 阶段验收标准

完成后应达到：

- prediction 不只是“执行”，而是“执行 + 汇总 + 导出 + 整理”
- 用户可以围绕同一次 prediction 做多轮 follow-up
- 产物路径和报告路径都可追溯

---

## Phase C：实时输入源预测（第三优先）

### 目标
把 Agent 从“文件型 prediction”推进到：

> **能接实时输入源，但仍保持 headless、可控、可停止。**

### C1. 摄像头预测

用户能直接说人话：

- 扫描可用摄像头
- 用某个摄像头开始预测
- 停止实时预测

### C2. RTSP 流预测

用户能直接说人话：

- 测一下这个 RTSP 地址能不能用
- 接这个视频流跑检测
- 停止流预测

### C3. 屏幕内容预测

用户能直接说人话：

- 对当前屏幕内容做检测
- 只录一段时间
- 保存结果

### C 阶段风险

这是整条支线里最容易变重的一段，因为它天然带来：

- 实时状态管理
- 长生命周期任务
- 启停控制
- 输出流与保存策略
- 更复杂的资源占用

所以这一阶段必须后置，不和 Phase A 并行抢资源。

### C 阶段验收标准

完成后应达到：

- 至少支持一个稳定的实时源类型先落地
- 能开始、能停止、能给状态
- 不会污染现有训练主线状态
- 不会让 agent 聊天层失去 grounded 约束

---

## 5. 推荐执行顺序

### 第 1 批（马上做）

先做数据治理里最有价值、最安全的一组：

1. 预览数据格式转换
2. 执行数据格式转换
3. 预览标签修改
4. 执行标签修改
5. 清理孤儿标签

### 第 2 批

继续做数据治理：

6. 生成空标签 / 缺失标签
7. 按类别整理数据

### 第 3 批

再补 prediction 结果管理：

8. 导出预测报告
9. 输出目录与结果清单管理
10. 预测结果二次整理

### 第 4 批

最后才碰实时输入：

11. 摄像头预测
12. RTSP 流预测
13. 屏幕预测

---

## 6. 每一批都要怎么做

每一批都按同一条节奏推进：

1. **core 能力确认**
   - 先确认原项目已有 headless 可复用逻辑

2. **service 封装**
   - 先做服务层，不直接把 GUI 行为原样搬进 tool

3. **tool 暴露**
   - 统一输出结构
   - 统一错误包装
   - 统一 summary / warnings / artifacts / next_actions

4. **聊天接入**
   - 加路由
   - 加 grounded reply
   - 加确认边界

5. **本地测试**
   - 工具级测试
   - 路由测试
   - 必要的状态写回测试

6. **远端验证**
   - 同步到 `yolostudio`
   - 跑远端 smoke
   - 需要时补 live 对话回归

---

## 7. 回归门和保护措施

### 7.1 主线保护门

支线每推进一批，至少要重新确认：

- 训练主线不回退
- prediction 主线不回退
- 训练计划草案确认层不回退
- 已有 chaos / live 对话关键路径不回退

### 7.2 高风险动作保护门

以下能力必须强制要求 preview 或确认：

- 批量改标签
- 批量转格式
- 清理文件
- 批量写出到新目录
- 覆盖旧输出

### 7.3 实时任务保护门

实时类能力必须明确：

- 谁在运行
- 运行状态
- 怎么停止
- 结果保存到哪
- 出错如何恢复
